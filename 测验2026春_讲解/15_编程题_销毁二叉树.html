🗑️ 编程题：销毁二叉树（后序遍历）

第15题 · 编写递归函数DestroyTree，释放所有结点并返回删除结点数
✅ 答案：int DestroyTree(T) { if(T==NULL) return 0; int n=DestroyTree(T->Left)+DestroyTree(T->Right); free(T); return n+1; }

一、题目理解

1.1 问题描述

给定一棵二叉链存储的二叉树，编写递归函数 DestroyTree，要求：

  1. 释放树中所有结点占用的内存（调用free）
  2. 返回被删除的结点总数
  3. 必须使用后序遍历的框架（先删孩子，再删根）

1.2 数据结构

struct TreeNode { int Data; struct TreeNode *Left; // 左孩子指针 struct TreeNode *Right; // 右孩子指针 };

二、完整答案代码

2.1 DestroyTree函数

/* 后序遍历销毁二叉树，返回删除的结点总数 */ int DestroyTree(struct TreeNode *T) { if (T == NULL) // ── ① 递归基：空树返回0 return 0; // ── ② 后序遍历：先递归销毁左右子树 ── int n = DestroyTree(T->Left) // 销毁左子树，得左子树结点数 + DestroyTree(T->Right); // 销毁右子树，得右子树结点数 // ── ③ 此时左右孩子已全部释放，安全删除根 ── free(T); // 释放当前根结点 return n + 1; // ── ④ 返回：子树结点数 + 自身(1) ── }

2.2 代码极简版（一行核心逻辑）

int DestroyTree(struct TreeNode *T) { if (T == NULL) return 0; int n = DestroyTree(T->Left) + DestroyTree(T->Right); free(T); return n + 1; }

三、为什么必须用后序遍历？

3.1 核心原因：先删根就找不到孩子了！

假设有一棵简单二叉树： ┌───┐ │ A │ ← 根 └─┬─┘ / \ / \ ┌───┐ ┌───┐ │ B │ │ C │ ← 左右孩子 └───┘ └───┘ ╔════════════════════════════════════════════════════╗ ║ ❌ 错误做法：先序（先删根） ║ ╠════════════════════════════════════════════════════╣ ║ ║ ║ 第1步：free(A) ← 释放根结点 ║ ║ ║ ║ 第2步：想删B ... 但 A->Left 指向的内存已被释放/ ║ ║ 不可访问！此时 T->Left 是悬垂指针 ║ ║ (dangling pointer)，访问它 = 未定义行为！ ║ ║ ║ ║ 结果：B和C永远无法被释放 → 内存泄漏！ ║ ║ ║ ╠══════════════════════════════════════════════════════╣ ║ ✅ 正确做法：后序（先删孩子） ║ ╠══════════════════════════════════════════════════════╣ ║ ║ ║ 第1步：递归到B，B是叶子 → free(B) ║ ║ 第2步：递归到C，C是叶子 → free(C) ║ ║ 第3步：回到A，此时B和C都已释放 ║ ║ free(A) ← 安全！A已无"牵挂" ║ ║ ║ ║ 结果：所有结点正确释放，无内存泄漏 ✓ ║ ╚══════════════════════════════════════════════════════╝

3.2 三种遍历方式对比

遍历方式访问顺序能否用于销毁？原因
先序 NLR根→左→右 ❌ 不行 先删根导致Left/Right指针失效，找不到孩子
中序 LNR左→根→右 ❌ 不行 删完左子树后删根，根的Right指针失效，找不到右子树
后序 LRN左→右→根 ✅ 唯一可行 根最后删除，左右子树早已安全释放
🔴 一句话总结：要安全释放二叉树，必须先释放所有子树，最后释放根。而这恰好是后序遍历的定义——左→右→根。所以销毁二叉树必须用后序。

五、递归逻辑深入分析

5.1 返回值是如何计算的？

DestroyTree(结点) 的返回值 = 该结点及其所有后代结点的总数 递推公式： DestroyTree(NULL) = 0 （空树0个结点） DestroyTree(T) = DestroyTree(T->Left) （左子树的结点数） + DestroyTree(T->Right) （右子树的结点数） + 1 （T自身） 以结点②为例： DestroyTree(②) = DestroyTree(④) + DestroyTree(⑤) + 1 = 1 + 1 + 1 = 3 （子树②共有3个结点：④⑤②） 以根结点①为例： DestroyTree(①) = DestroyTree(②) + DestroyTree(③) + 1 = 3 + 3 + 1 = 7 （整棵树共7个结点）✓

5.2 递归执行流程图

DestroyTree(①) │ ├─ n_left = DestroyTree(②) │ │ │ ├─ n_ll = DestroyTree(④) │ │ └─ ④是叶子: free(④), return 1 │ │ │ ├─ n_lr = DestroyTree(⑤) │ │ └─ ⑤是叶子: free(⑤), return 1 │ │ │ └─ free(②), return 1+1+1 = 3 │ ├─ n_right = DestroyTree(③) │ │ │ ├─ n_rl = DestroyTree(⑥) │ │ └─ ⑥是叶子: free(⑥), return 1 │ │ │ ├─ n_rr = DestroyTree(⑦) │ │ └─ ⑦是叶子: free(⑦), return 1 │ │ │ └─ free(③), return 1+1+1 = 3 │ └─ free(①), return 3+3+1 = 7 要点：free(T)总是在左右子树都递归返回后才执行 → 天然的后序遍历顺序 ✓

六、常见错误 & 变体

⚠️ 错误1：忘记递归基。如果漏掉 if (T == NULL) return 0;，对叶子结点的孩子（NULL）调用时会发生段错误。
⚠️ 错误2：使用先序或中序。如果写成先序（先free(T)再递归），free(T)后T->Left和T->Right是悬垂指针，后续访问导致未定义行为。
⚠️ 错误3：返回值计算错误。return n+1 中的"+1"是当前结点自身。不要写成 return n（漏掉自身）或 return n+2。
⚠️ 错误4：free后未置NULL。在DestroyTree内部，free(T)后T是局部变量，函数返回后自动销毁，不需要置NULL。但如果有全局根指针，调用者应在DestroyTree后将根指针置为NULL，避免外部悬垂指针。

6.1 调用示例

int main() { struct TreeNode *root = BuildTree(); // 假设已构建好树 int count = DestroyTree(root); // 销毁整棵树 root = NULL; // ★ 重要：将根指针置NULL！ printf("共删除 %d 个结点\n", count); // 输出：共删除 7 个结点 return 0; }

七、与其他树操作的对比

树操作推荐遍历原因
销毁二叉树 后序 LRN 必须先删孩子才能安全删根
复制二叉树 先序 NLR 先创建根，再递归复制左右子树
求树的高度 后序 LRN 先知道左右子树高度，才能算根的高度
输出BST有序序列 中序 LNR BST中序 = 升序输出
查找结点 先序/中序/后序均可 取决于查找策略（DFS/BFS）
统计结点数 后序 LRN 先统计子树再+1，逻辑最自然
💡 规律：需要自底向上汇总信息的操作（销毁、求高度、求结点数）通常用后序；需要自顶向下传递信息的操作（复制、先序遍历）通常用先序；需要按顺序处理的操作（BST输出）用中序。

八、考点总结

💡 考点1：后序遍历的应用场景。这是后序遍历最经典的用途——安全释放二叉树。面试中高频出现"为什么销毁二叉树必须用后序"。
💡 考点2：递归返回值设计。函数不仅完成销毁，还返回结点总数。这展示了"递归函数可以有副作用（free）+ 返回值（计数）"的设计模式。
💡 考点3：递归基的重要性。空树判断是递归的"终止条件"，缺了它递归永远不会停下来（或者说对NULL解引用会直接崩溃）。
💡 记忆口诀："删树必须后序走，先删孩子再删根。若是先序或中序，孩子丢了没处寻。"
