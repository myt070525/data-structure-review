🌲 编程题：DestroyTree销毁二叉树

第29题 · 后序遍历释放所有结点并返回总数
✅ 完整代码 · 后序遍历 · 返回结点个数

一、题目理解：我们要做什么？

1.1 问题描述

给定一棵链式存储的二叉树（每个结点有Data、Left指针、Right指针），编写函数 DestroyTree， 递归地销毁整棵树——释放每一个结点的内存，并返回被销毁的结点总数。

1.2 核心任务

  1. 遍历所有结点：不能遗漏任何一个结点。
  2. 释放内存：对每个结点调用 free()。
  3. 保证安全：在释放父结点之前，必须先释放其孩子结点。否则丢失孩子地址 → 内存泄漏。
  4. 统计个数：返回总共释放了多少个结点。

1.3 二叉树结点结构

typedef struct TreeNode { int Data; struct TreeNode *Left; // 左孩子指针 struct TreeNode *Right; // 右孩子指针 } TreeNode;

二、为什么必须用后序遍历？——这是本题最核心的考点

2.1 三种遍历顺序对比

遍历方式访问顺序根结点何时free？能否安全销毁整棵树？
先序 根 → 左子树 → 右子树 第一步就free根 ❌ 不行！根被释放后，Left和Right指针丢失，孩子结点地址再也找不到了 → 内存泄漏
中序 左子树 → 根 → 右子树 处理完左子树后free根 ❌ 不行！free根之后Right指针丢失，右子树全部泄漏
后序 ✅ 左子树 → 右子树 → 根 最后才free根 ✅ 完美！释放根时左右孩子早已释放完毕，没有任何指针丢失

2.2 形象类比：拆楼

想象你要拆除一栋三层楼房（二叉树）：

  1. 先序拆法：先炸掉一楼（根）→ 整栋楼塌了，二楼三楼砸在废墟里，你再也找不到它们在哪 → 内存泄漏。
  2. 中序拆法：先拆左边的二楼 → 再炸一楼（根）→ 右边的二楼三楼随着倒塌也找不到了 → 部分泄漏。
  3. 后序拆法 ✅：先拆完所有上层（左子树+右子树），最后拆一楼（根）→ 每一层都妥善处理，零泄漏。

2.3 代码层面的解释

// 错误示例：先序销毁（会内存泄漏！） int BadDestroy(TreeNode *T) { if (T == NULL) return 0; TreeNode *left = T->Left; // 赶在free前保存孩子地址 TreeNode *right = T->Right; free(T); // ❌ 先释放根 return 1 + BadDestroy(left) + BadDestroy(right); } // 这种写法虽然预先保存了left/right，但破坏了递归的优雅性。 // 标准答案直接用后序遍历，无需额外变量。
💡 核心结论：后序遍历是销毁树状结构的唯一安全方式。先释放孩子，再释放父结点——这是树结构内存管理的基本原则。

三、完整代码 & 逐行解析

3.1 DestroyTree函数

int DestroyTree(struct TreeNode *T) { int n = 0; // 局部计数器，初始为0 if (T == NULL) // 递归终止条件：空树/空子树 return 0; // 空树没有结点可释放，返回0 // ─── 后序遍历核心：左 → 右 → 根 ─── n += DestroyTree(T->Left); // ① 先递归销毁左子树，累加左子树结点数 n += DestroyTree(T->Right); // ② 再递归销毁右子树，累加右子树结点数 free(T); // ③ 最后释放当前根结点 return n + 1; // ④ 返回：左右子树结点数 + 当前结点(1) }

3.2 逐行详解

代码作用关键理解
int n = 0; 初始化局部计数器。每次递归调用都有自己独立的n。 每个结点的栈帧中n都从0开始，最终通过return层层累加。
if (T == NULL) return 0; 递归基（终止条件）。当走到叶子结点的孩子（NULL）时停止。
返回0表示"空子树有0个结点需要释放"。 		这是递归函数的"触底"条件。没有它递归就会无限调用直到栈溢出。
n += DestroyTree(T->Left); 递归处理左子树。函数返回左子树中被销毁的结点总数。
这一步结束后，T->Left指向的整棵左子树已经全部被free。 		此后T->Left虽然还保留着地址值，但指向的内存已经无效（悬空指针）。不过我们马上也要free(T)了，不影响。
n += DestroyTree(T->Right); 递归处理右子树。同理返回右子树结点总数。 此时左右子树都已销毁，只剩下根结点T还未释放。
free(T); 释放根结点T。此时T的孩子早已释放完毕，没有任何依赖。 这就是后序的关键！free(T)在所有递归调用之后执行。
return n + 1; 返回本层销毁的结点总数 = 左子树结点数 + 右子树结点数 + 当前根(1)。
这个值被上层调用者累加到它的n中。 		+1非常关键——不能只返回n，否则所有结点的计数都会丢失。

A 层级： / \ A (根) B C B C (第二层) / \ / \ D E F G (叶子层) D E F G 结点总数 = 7 后序销毁顺序：D → E → B → F → G → C → A （叶子先于分支结点，分支结点先于根）

六、错误遍历顺序的后果演示

6.1 先序销毁：free根 → 丢失孩子 → 内存泄漏

// 错误代码：先序销毁 int BadDestroy_PreOrder(TreeNode *T) { if (T == NULL) return 0; free(T); // ❌ 先释放根！ // 下面两行试图访问T->Left和T->Right → 但T已被释放！ // 这是"use-after-free"——未定义行为（UB） return 1 + BadDestroy_PreOrder(T->Left) // 💥 崩溃或读到垃圾 + BadDestroy_PreOrder(T->Right); }
执行过程： DestroyTree(A) ├─ free(A) ← A的内存被标记为"可用"，但A->Left/A->Right可能已被覆盖 ├─ 访问 T->Left ← 💥 T已经free了！T->Left现在是垃圾值 └─ 结果：要么崩溃，要么递归到随机地址，B和C的整棵子树永远丢失 内存泄漏数量：B,C,D,E,F,G共6个结点无法释放！

6.2 中序销毁：free根后右子树丢失

// 错误代码：中序销毁 int BadDestroy_InOrder(TreeNode *T) { if (T == NULL) return 0; int n = BadDestroy_InOrder(T->Left); // ① 左子树OK free(T); // ② ❌ 释放根 n += BadDestroy_InOrder(T->Right); // ③ 💥 T已free！ return n + 1; }
执行过程（以结点B为例）： DestroyTree(B) ├─ ① 递归销毁D（OK）→ 返回1 ├─ ② free(B) ← B被释放 ├─ ③ 访问T->Right ← 💥 B已free，B->Right（指向E）可能已丢失 └─ 结果：E以及E的所有后代永远泄漏 对A来说：左子树B被部分泄漏（E泄漏），右子树C全部泄漏（因为free(A)后C丢失）

6.3 对比总结

遍历方式泄漏的结点根本原因
先序 几乎整棵树（除了根） 第一步就free根，Left和Right指针立即失效
中序 每个结点的右子树 free根后再访问Right，指针已失效
后序 ✅ 0个！ free根时左右孩子早已安全释放完毕
🔴 核心教训：在C语言中，一旦对指针执行了 free()，该指针指向的内存就不再属于你。之后任何通过该指针访问成员（如T->Left、T->Right）的行为都是未定义行为。因此，在free一个结点之前，必须确保它的所有孩子结点都已经被释放——而这正是后序遍历保证的。

七、边界情况分析

情况1：空树（T == NULL）

DestroyTree(NULL); → if (T == NULL) return 0; 直接返回0，无任何操作。✅

情况2：只有一个结点（只有根）

调用 DestroyTree(根) ├─ n += DestroyTree(NULL) → 0 (无左孩子) ├─ n += DestroyTree(NULL) → 0 (无右孩子) ├─ free(根) └─ return 0 + 1 = 1 ✅

情况3：单链树（每个结点只有一个孩子，退化成链表）

A → B → C → NULL （每个结点只有Right孩子） 后序销毁：C → B → A ├─ DestroyTree(A) → 调用DestroyTree(B) │ └─ DestroyTree(B) → 调用DestroyTree(C) │ └─ DestroyTree(C) → no children → free(C) → return 1 │ └─ free(B) → return 2 └─ free(A) → return 3 ✅ 即使退化成链表，后序遍历依然正确工作。 注意：这里每个结点只有Right孩子，但代码会先检查Left（NULL返回0），再Right。

情况4：完全二叉树

深度为3的满二叉树（共15个结点） 后序销毁从最底层叶子开始，逐层向上。 每一层都在其上层被释放之前完全释放。✅
💡 总结：无论树的形态如何（空树、单结点、链表状、完全树），后序遍历销毁策略都天然正确，无需任何特殊处理。这是递归算法的优雅之处。

八、常见错误 & 考点总结

🔴 致命错误1：先free(T)再递归子树（先序/中序错误）。
这是本题最核心的考点。free(T)后T->Left和T->Right是未定义行为，会导致崩溃或内存泄漏。
正确做法：free(T)必须是递归调用之后的最后一步操作。
记忆口诀："先孩子，后自己；先递归，后free。"
⚠️ 错误2：return n 而忘记 +1。
如果return n而不是return n+1，每个结点都不会被计入总数，最终返回0。
n累积的是子树结点数，+1是把当前结点自己算进去。
⚠️ 错误3：忘了递归基（T==NULL的判断）。
没有终止条件，递归会一直调用到栈溢出（Stack Overflow）。
叶结点的孩子是NULL → 必须return 0作为递归的"触底"。
⚠️ 错误4：写成void函数不返回个数。
题目明确要求返回删除个数。虽然销毁本身不需要返回值，但统计结点数是考试要求。
⚠️ 错误5：混淆"后序遍历"和"后序销毁"。
后序遍历的一般模式是"左→右→根"，本题的销毁操作放在"根"的位置（即递归调用之后）。
不要把free(T)放在递归调用之前。
💡 考试心法：
这道题的本质是考察三个知识点的融合——
① 二叉树的后序遍历（左→右→根的递归框架）
② C语言内存管理（malloc/free的配对，防止内存泄漏）
③ 递归计数（通过return值层层累加结点总数）
理解"为什么必须是后序"比记住代码更重要——因为这是所有树形结构内存管理的基础原则。
💡 记忆口诀："递归先判空返回零，左子右子依次行。子树数目累加到n，释放当前根结点。返回n加一不能忘，后序销毁最安全。"

九、拓展思考：如果不用递归呢？

也可以使用迭代 + 栈来模拟后序遍历销毁，但代码会复杂很多：

// 迭代版本（仅供参考，考试不要求） int DestroyTree_Iterative(TreeNode *root) { if (root == NULL) return 0; TreeNode *stack[1000]; // 假设深度不超过1000 int top = -1, count = 0; TreeNode *cur = root, *last = NULL; while (cur || top >= 0) { while (cur) { stack[++top] = cur; cur = cur->Left; } cur = stack[top]; if (cur->Right == NULL || cur->Right == last) { last = cur; free(cur); count++; top--; cur = NULL; } else cur = cur->Right; } return count; }
💡 递归版本简洁优雅，是考试的标准答案。迭代版本虽然避免了递归栈开销，但代码量大了3倍以上。在考试中，递归后序遍历就是最优解。
